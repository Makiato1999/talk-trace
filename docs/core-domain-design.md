# TalkTrace 核心领域设计

## 核心对象

TalkTrace 当前只需要理解五个主要对象：

| 对象 | 含义 |
| --- | --- |
| Technical Talk | 一场技术演讲 |
| Source | 演讲材料，例如 transcript、字幕或 slides |
| Engineering Note | 从演讲中整理出的结构化技术笔记 |
| Evidence | 笔记内容与演讲原始材料之间的关联 |
| Revision | 笔记的一个版本 |

另外还有一个系统对象：

| 对象 | 含义 |
| --- | --- |
| Pipeline Run | AI 根据指定输入生成一次候选笔记的过程和结果 |

## Engineering Note Protocol

Protocol 是笔记的内容范式。它采用：

> 固定骨架 + 可选模块 + 组合规则

### 固定骨架

每篇笔记都应具备最基本的信息：

- 标题；
- 演讲与讲者信息；
- 内容概览；
- 关键收获；
- 来源与证据定位。

### 可选模块

根据演讲内容，自适应选择：

- 问题与背景；
- 核心观点；
- 原理与机制；
- 架构或流程；
- 示例与演示；
- 方案比较；
- 实践建议；
- 限制与注意事项；
- 延伸资源。

不是每篇笔记都需要所有模块，也不应该为了填满模板而生成没有价值的内容。

### 组合规则

Protocol 需要规定：

- 什么情况下使用哪个模块；
- 模块之间如何组织；
- 哪些内容必须关联证据；
- 如何区分演讲者观点、事实陈述和 AI 整理；
- 如何表达不确定、条件和限制；
- 哪些内容不能被 AI 擅自补充。

因此，Protocol 定义的是“合格的工程笔记是什么”，而不是具体怎样调用模型。

## Pipeline

Pipeline 是 TalkTrace 的 AI 总结系统。

```text
Technical Talk
transcript / slides
        |
        v
Understand
识别主题和知识结构
        |
        v
Compose
选择模块并生成草稿
        |
        v
Validate ----------------------> Engineering Note
检查结构、证据和遗漏     通过       候选笔记
        |
        | 发现问题
        v
Revise
根据问题修订草稿
        |
        +----------------------> Validate
```

这张图表示 AI 的生成工作流，不是笔记的版本状态机。

最基本的工作方式是：

```text
理解演讲
  -> 判断笔记需要哪些模块
  -> 生成草稿
  -> 检查结构、证据和遗漏
  -> 修订草稿
  -> 输出候选笔记
```

这个过程可以叫做 `Draft -> Validate -> Revise` Loop。

目前不必把它设计成复杂 Agent。以后如果 Pipeline 需要搜索原文、调用工具或动态规划，再考虑 ReAct 等实现范式。

## Revision

这里的 Revision 是“笔记版本”，不是 ReAct。

```text
AI 生成版本
  -> 用户检查和修改
  -> 保存版本
  -> 继续编辑或重新生成
```

基本规则：

- AI 生成的版本可以被用户修改；
- 保存后可以继续产生新版本；
- 是否长期保留完整版本历史，留到数据设计阶段决定。

Pipeline 内部也会修订草稿，但那是生成过程，不等同于用户保存的笔记版本。

## Evidence

Evidence 用于回答：

> 这条笔记来自演讲的哪个位置？

优先为以下内容保留证据：

- 核心观点；
- 技术原理；
- 重要数据或判断；
- 示例；
- 实践建议；
- 限制和注意事项。

证据用于忠实呈现演讲内容，也方便用户快速回到 transcript、时间点或 slide。它不代表演讲者的说法一定正确。

## 当前核心规则

1. AI 负责生成，用户负责检查和修改。
2. 笔记按知识结构组织，不按演讲时间线复述。
3. Protocol 决定笔记结构，Pipeline 决定生成过程。
4. 重要内容尽量能回到原始演讲材料。
5. 不能把 AI 的补充写成演讲者的观点。
6. 演讲中的条件、限制和不确定性需要保留。

## 暂时不设计

当前阶段先不展开：

- Skills；
- 具体模型和 Prompt；
- ReAct Agent；
- 数据库表结构；
- 完整状态机；
- 聚合、领域事件和微服务边界；
- 多人协作和复杂权限。

下一步应该先用几类真实技术演讲，验证 Engineering Note Protocol 的固定骨架、可选模块和组合规则。
